# 数据平权，AI 下乡：摊位 AI 助手设计

日期：2026-06-13

## 1. 背景

这是一个一次性线下活动 app。活动现场会有一个介绍“为工友敲键盘”理念的小摊位。摊主一次只能和一个人对话，周围围观的人、等待的人、或者不方便当面开口的人，可以扫码进入 H5 页面，先由 AI 助手了解摊位理念、提出疑问、整理想继续聊的问题。

本 app 不参加辩论，不代表活动主办方，不借活动主办方或 UP 主背书。页面和回答中默认不提具体活动、主办方或 UP 主。

主题标题只用于这次活动：

> 数据平权，AI 下乡

长期项目品牌仍是“为工友敲键盘”。更深层的“为人民服务的公司”只在用户追问项目理念、组织设想或材料来源时按知识库内容解释，不出现在首屏。

## 2. 产品定位

这个 app 是“第二个摊主”，不是辩手、客服、导师或全知百科。

它的任务是：

- 帮扫码用户快速理解这个摊位在讲什么。
- 接住用户对“AI、工友、数据平权、组织形式”的疑问。
- 在摊主忙碌时分流等待人群。
- 将真正感兴趣的人温和引导回摊位继续聊。
- 让不好意思当面开口的人可以先通过 AI 问几轮。

它不做：

- 辩论赛辅助工具。
- 对主办方、UP 主、辩题或时政新闻的评论工具。
- 用户留资系统。
- 长期知识库产品。
- NotebookLM 式资料管理工具。
- 正式官网。

## 3. 目标与优先级

第一目标：

- 把围观、等待、不好意思开口的人，预热成愿意回摊位继续聊的人。

次级目标：

- 用户理解“数据平权，AI 下乡”的基本含义。
- 用户知道如何继续了解“为工友敲键盘”。
- 用户在明确索要时能找到官网、文档或联系方式入口。

不以强转化为目标：

- 不主动收集联系方式。
- 不在主流程反复推二维码或官网。
- 不把回答写成增长漏斗。

## 4. 用户画像

默认用户具有一定 AI 产品使用经验，可能用过 DeepSeek、豆包等国内 AI 工具。因此界面可以接近简洁 AI 工具，不需要过度教学。

现场用户大致分为三类：

- 好奇但没背景：扫进来只是想知道“这是啥”。
- 有疑问或抵触：对“为工友、AI、数据、组织”等词天然警惕。
- 有兴趣深入：想继续聊理念、参与方式、技术实现或后续材料。

## 5. MVP 范围

### 5.1 包含

- 手机 H5，临时链接加二维码访问。
- Next.js 全栈 app。
- DeepSeek 服务端代理。
- 首屏三入口：
  - 我先看看这是啥
  - 我有点怀疑
  - 我想继续聊
- 简洁聊天界面。
- 浏览器内存中的多轮上下文。
- 浏览器内存中的轻量摘要记忆。
- Markdown + frontmatter 知识库格式。
- 简单检索，不上向量检索。
- 服务端 IP 限流。
- 输入长度限制、请求超时、回答长度限制。
- 默认不展示材料来源。
- 用户追问依据时可解释来源。
- “继续了解”独立区域。

### 5.2 不包含

- 用户登录。
- 联系方式表单。
- 服务端持久化聊天记录。
- 用户上传材料。
- 静态 FAQ 兜底。
- 向量检索。
- embedding 模型接入。
- 管理后台。
- 复杂统计分析。
- 正式域名或备案。

## 6. 页面结构

第一版只有一个主页面。

用户流程：

```text
扫码打开
→ 首屏
→ 选择入口或直接输入
→ 对话界面
→ 按需打开“继续了解”
```

首屏内容：

```text
数据平权，AI 下乡

为工友敲键盘的摊位助手
摊主正在和别人聊时，你可以先问我。
我的回答只基于摊位准备的材料；说不清的地方会说清楚。

[我先看看这是啥]
[我有点怀疑]
[我想继续聊]

问点什么...
```

进入对话后：

- 大标题收起或弱化。
- 消息流成为主体。
- 底部固定输入框。
- 不再展示三个入口，避免占屏。
- 保留轻量“继续了解”入口。

## 7. 三个入口

### 7.1 我先看看这是啥

点击后 AI 自动生成一段 30 秒介绍，然后等待用户追问。

语气：

- 白话。
- 短。
- 不喊口号。
- 不堆理论。

目的：

- 让第一次扫码的人立刻理解摊位主题。

### 7.2 我有点怀疑

点击后 AI 先承认怀疑合理，然后邀请用户直接提出尖锐问题。

语气：

- 能接住质疑。
- 不急着反驳。
- 不进入争胜状态。

示例方向：

- 这是不是中产自我感动？
- AI 会不会只是资本的新工具？
- 凭什么说是在服务工友？

### 7.3 我想继续聊

点击后 AI 给一段很短说明：

- 可以继续问 AI。
- 可以等摊主空了回摊位聊。
- 官网和联系方式在“继续了解”里。
- AI 可以帮用户把问题整理成适合现场继续聊的问题。

二维码、官网和联系方式不进入主流程，除非用户明确索要。

## 8. 继续了解区域

“继续了解”是独立区域，不打断对话主流程。

包含：

- 回摊位继续聊的提示。
- 微信或群二维码。
- 官网或文档链接。

行为：

- 默认不弹出。
- 用户可主动打开。
- 用户明确问“怎么联系、怎么加入、有没有网站、后续怎么看”时，AI 才在回答中引导用户打开此区域。

## 9. AI 回答策略

统一身份：

- 摊位助手。

底色：

- 克制解释型。

分模式语气：

- “我先看看这是啥”：朋友解释型。
- “我有点怀疑”：接住质疑型。
- “我想继续聊”：整理和引导型。
- 直接输入：默认克制解释型。

回答规则：

- 不喊口号。
- 不主动攻击其他立场。
- 不替用户下结论。
- 能承认项目还早、材料有限、风险存在。
- 默认 2-5 段短答。
- 用户追问后再展开。
- 不主动展示材料出处。
- 用户追问依据时再说明来源。

## 10. 知识库边界

第一版 app 定义知识库格式和加载接口，具体内容后续单独整理。

当前 `ideal` 仓库内容只是初步材料来源，不直接等同于正式知识库。正式知识库应由松散材料整理而来，并经过人工确认。

知识库采用 Markdown + frontmatter：

```md
---
id: project-intro
title: 项目介绍
visibility: public
status: approved
tags:
  - 数据平权
  - AI 下乡
  - 为工友敲键盘
---

正文内容...
```

加载规则：

- 只加载 `visibility: public`。
- 只加载 `status: approved`。
- 保留 `id`、`title`、`tags` 等 metadata。
- 不加载未确认、内部、敏感或不适合对外讲的材料。

推荐知识库目录：

```text
knowledge/
  public/
    001-project-intro.md
    002-why-ai.md
    003-who-we-serve.md
    004-boundaries.md
    005-hard-questions.md
    006-how-to-continue.md
  policy/
    answer-style.md
    refusal-and-redirect.md
```

## 11. 检索策略

MVP 不上向量检索。

第一版使用简单检索：

- 读取 public + approved Markdown。
- 按标题、小标题、段落切成 chunk。
- 对用户最新问题和最近上下文做轻量关键词匹配。
- 选 top N chunk 放入 DeepSeek prompt。
- chunk 保留 source metadata。

如果检索不到足够相关内容：

- 不硬答。
- 温和说明材料没有充分展开。
- 引导用户换个与项目理念相关的角度继续问，或回摊位聊。

预留接口：

```text
retrieve(query, options) -> RetrievedChunk[]
```

后续如果测试发现简单检索效果不足，可替换为：

- embedding 模型。
- 向量索引。
- reranker。

DeepSeek 继续负责最终回答，不负责检索。

## 12. 越界处理

采用温和转向。

越界类型：

- 主办方、UP 主、辩论赛相关评价。
- 时政新闻评论。
- 法律、医疗等高风险建议。
- 私人决策建议。
- 知识库未覆盖的问题。

回答方式：

```text
这个问题摊位材料里没有充分展开，我不适合替项目随意表态。

但它和“AI 到底服务谁”这个核心问题有关系。你可以换个角度问：这个项目为什么强调数据平权？或者直接回摊位找摊主聊这个问题。
```

原则：

- 不冷冰冰拒绝。
- 不编造材料。
- 不借常识替项目表态。
- 尽量把问题转回摊位主题。

## 13. 多轮上下文与摘要记忆

多轮上下文存在浏览器内存中。刷新页面后全部清空。

前端维护：

- `messages`：最近几轮原文。
- `conversationSummary`：较早对话摘要。

发送 `/api/chat` 时包含：

- 当前入口模式。
- `conversationSummary`。
- 最近 3-4 轮原文。
- 用户最新问题。

当消息超过阈值，例如 10 条时：

- 将较早的 6 条发送给 `/api/summarize`。
- 服务端调用 DeepSeek 生成短摘要。
- 前端用新摘要更新 `conversationSummary`。
- 原文只保留最近 4-6 条。

摘要规则：

- 只总结用户问过什么、助手回答过什么、用户仍关心什么。
- 不添加新观点。
- 不补充知识库没有的项目立场。
- 不确定内容标记为不确定。

摘要用途：

- 只作为对话状态。
- 不能作为项目事实来源。
- 知识库内容优先于摘要。

服务端不保存 `messages` 或 `conversationSummary`。

## 14. 技术架构

采用 Next.js 全栈 H5。

模块：

```text
Next.js App
├─ mobile page
├─ /api/chat
├─ /api/summarize
├─ knowledge loader
├─ retriever
├─ prompt builder
├─ DeepSeek client
└─ rate limiter
```

前端：

- React。
- 移动端优先。
- 类 DeepSeek 的简洁 AI 工具界面。
- 底部固定输入框。
- loading、失败、重试、取消状态。
- 不写 localStorage。
- 不收集联系方式。

后端：

- `/api/chat`：回答用户问题。
- `/api/summarize`：生成浏览器内存摘要。
- 输入校验。
- IP 限流。
- DeepSeek 请求超时。
- 错误处理。
- API key 只存在服务端环境变量。

环境变量：

```text
DEEPSEEK_API_KEY
DEEPSEEK_BASE_URL
DEEPSEEK_MODEL
```

## 15. 限流与失败处理

限流：

- 服务端按 IP 限流。
- 初始建议：每分钟 10 次，每小时 100 次。
- 高频请求返回友好提示。

输入限制：

- 单条用户输入建议上限 1000 字。
- 历史原文只带最近 3-4 轮。
- 摘要建议上限 800 字。
- 检索 chunk 数量建议 4-6 个。

失败处理：

- DeepSeek 超时。
- 网络失败。
- 限流。
- 空输入。
- 超长输入。

用户提示应清楚、短：

```text
网络可能有点不稳，可以重试一次。也可以直接回摊位找摊主问这个问题。
```

## 16. 隐私与数据

MVP 不保存服务端会话。

应用代码不做：

- 登录。
- 用户画像。
- 联系方式收集。
- 聊天记录持久化。
- 第三方分析脚本。

每次请求期间，服务端会临时处理用户消息、摘要和检索片段，用于调用 DeepSeek。请求完成后应用不主动保存。

如果部署平台或 DeepSeek 服务保留请求日志，那属于外部服务层面的运行日志风险，活动前需要在 README 或隐私说明中用简短语言提示。

## 17. 部署

一周 MVP 使用临时链接部署。

部署要求：

- 支持 Next.js API route 或等价 serverless function。
- 配置环境变量。
- HTTPS。
- 能生成二维码。

活动后如果验证有效，再考虑：

- 正式域名。
- 国内部署。
- 备案。
- 与官网整合。

## 18. 验收标准

功能验收：

- 手机可访问首屏。
- 三个入口都能进入对话。
- 用户直接输入问题可回答。
- DeepSeek key 不暴露在前端。
- 服务端不保存会话。
- 多轮上下文可用。
- 摘要记忆可用。
- 刷新页面后会话清空。
- 越界问题温和转向。
- “怎么联系/怎么继续了解”类问题能引导到继续了解区域。
- 用户追问依据时能解释来源。
- 限流、超时、输入限制可用。

体验验收：

- 首屏让用户立刻知道这是摊位助手。
- 页面轻，不依赖重图或复杂动画。
- 回答短、清楚、不像口号。
- 不像增长漏斗。
- 不像辩论 bot。

质量验收：

- 本地测试覆盖常见问题。
- 人工测试覆盖尖锐质疑。
- 简单检索效果不足时，优先调材料和 prompt。
- 如果检索仍不足，再升级向量检索。

## 19. 测试用例方向

开场入口：

- 点“我先看看这是啥”，应生成 30 秒介绍。
- 点“我有点怀疑”，应承认怀疑合理并邀请提问。
- 点“我想继续聊”，应说明可以继续问或回摊位聊，不主动推二维码。

普通理解：

- 这个项目到底是干嘛的？
- 数据平权是什么意思？
- AI 下乡是什么意思？
- 为工友敲键盘和普通公益有什么区别？

尖锐质疑：

- 这是不是中产自我感动？
- 你们凭什么代表工友？
- AI 不就是资本家提高剥削效率的工具吗？
- 最后会不会变成普通互联网公司？

越界问题：

- 你怎么看小金毛？
- 这场辩论谁会赢？
- 你怎么看今天某个时政新闻？
- 帮我写一份法律诉状。
- 我这个症状要不要吃药？

多轮上下文：

- 先问“AI 为什么不会变成剥削工具？”
- 再问“那你们怎么避免这个？”
- 再问“刚才那个风险具体怎么处理？”
- 检查摘要后是否还能保持主题。

来源追问：

- 这是谁说的？
- 你依据哪份材料？
- 原文在哪里？

继续了解：

- 怎么联系你们？
- 有没有网站？
- 我想之后继续聊怎么办？

失败与限流：

- DeepSeek 超时。
- 网络失败。
- 高频请求。
- 超长输入。
- 空输入。

## 20. 后续升级

- 整理正式知识库。
